# Summary: CORS Headers on Error Responses Fix

This document summarizes the changes made to address the issue of missing CORS headers on error responses generated by SRouter.

## 1. Problem

When a route handler (especially a generic handler) returned an error (e.g., `router.HTTPError` or any other `error`), the resulting HTTP error response (e.g., 400 Bad Request, 500 Internal Server Error) sent to the client was missing necessary CORS headers (`Access-Control-Allow-Origin`, `Access-Control-Allow-Credentials`, etc.).

This occurred even if CORS middleware was correctly configured for the route and the preflight `OPTIONS` request succeeded. Browsers would then block frontend JavaScript (on a different origin) from reading the error response details due to CORS policy violations.

## 2. Cause

The root cause was the sequence of execution in the request/response cycle:

1.  Incoming request flows through the middleware chain (including CORS middleware).
2.  CORS middleware would set appropriate headers on the `http.ResponseWriter` *before* calling the next handler in the chain. It expects to potentially modify headers again *after* the handler returns successfully.
3.  If the route handler returned an `error`, internal error handling functions (`handleError`, `recoveryMiddleware`, etc.) would take over.
4.  These error handlers previously wrote the error response (status code, headers like `Content-Type`, JSON body) *directly* to the `http.ResponseWriter`.
5.  This direct writing bypassed the return path of the middleware chain. Middleware like CORS did not get a chance to inspect the final (error) status code or add/modify headers on the response before it was sent.

## 3. Solution: Delayed Response Writing

The implemented solution refactors the response writing mechanism to ensure all middleware has a chance to run before the final response (success or error) is sent.

1.  **`responseStateWriter` Introduced (`pkg/router/router.go`):**
    *   A new internal `http.ResponseWriter` wrapper (`responseStateWriter`) was created.
    *   This wrapper intercepts calls to `WriteHeader` and `Write`.
    *   `WriteHeader`: Captures the intended status code but *does not* write it to the underlying connection immediately.
    *   `Write`: Buffers the *successful* response body in memory (`successBody *bytes.Buffer`) instead of writing it immediately. It captures an implicit 200 OK status if `WriteHeader` wasn't called first.
    *   `setError`: A new method was added to capture the intended error status code and the pre-rendered error body (`errorBody []byte`). It prevents further writes to the success buffer.

2.  **Modified `ServeHTTP` (`pkg/router/router.go`):**
    *   The main `ServeHTTP` method now wraps the original `http.ResponseWriter` with `responseStateWriter`.
    *   This wrapped writer is passed down through the middleware chain and to the handler.
    *   After `r.router.ServeHTTP(finalWriter, req)` returns (meaning all middleware and the handler have executed), a final block in `ServeHTTP` inspects the state of `responseStateWriter`.
    *   It determines the final status code (captured explicitly or defaulted to 200).
    *   It writes the final status code to the *original* `http.ResponseWriter`.
    *   It then writes either the buffered `errorBody` (if `setError` was called) or the buffered `successBody` to the *original* `http.ResponseWriter`.

3.  **Updated Error Handling Paths:**
    *   `handleError` (`pkg/router/router.go`): Modified to no longer accept `http.ResponseWriter`. It now determines the status code, renders the JSON error body using `renderJSONErrorBody`, logs the error, and returns the `(int, []byte)` status and body.
    *   `RegisterGenericRoute` wrapper (`pkg/router/route.go`): When the user handler returns an error, or if decoding/encoding/sanitizing fails, it now calls the modified `handleError` to get the status/body and then calls `stateW.setError(status, body)` on the wrapped writer.
    *   `recoveryMiddleware`, `timeoutMiddleware`, `authRequiredMiddleware` (`pkg/router/router.go`): Updated to call `handleError` and then `stateW.setError(status, body)` when they need to generate an error response, instead of writing directly.

This ensures that the actual writing to the underlying connection happens only once at the very end of `ServeHTTP`, *after* all middleware (including CORS) has had the opportunity to inspect the request, run the handler (via `next.ServeHTTP`), and potentially modify the headers on the `responseStateWriter` based on the final intended status code.

## 4. Performance Impact

This refactoring introduces some performance considerations:

*   **Buffering:** The most significant change is that successful response bodies are now buffered entirely in memory (`responseStateWriter.successBody`) before being written to the connection.
    *   **Impact:** For small-to-medium responses, the overhead is likely negligible. For very large responses, this increases memory usage and potentially adds latency compared to directly streaming the response, as the full body must be generated before sending begins.
*   **Wrapper Overhead:** Introducing the `responseStateWriter` adds a small amount of overhead for each request due to the wrapper methods and mutex locking within them (though locks are brief).
*   **Error Path:** The error path involves rendering the JSON body via `handleError` and storing it in `errorBody`. Since error responses are typically small, the performance impact here is minimal.

**Overall:** The performance impact is expected to be low for typical API use cases involving reasonably sized responses. However, applications streaming very large response bodies might observe increased memory consumption and slightly higher latency due to the buffering. This trade-off was made to achieve the architecturally cleaner solution of allowing middleware to process the final response state correctly.
